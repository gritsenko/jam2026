# Онлайн-редактор геймдизайна: минимальный приватный деплой

**Статус:** планируемое (спроектировано, в коде/инфре ещё нет). Цель — поднять уже
реализованный [редактор геймдизайна](../backlog/design-editor.md) на онлайн-хосте, чтобы
править конфиги и прогонять ботов **с любого устройства**, по образцу
[онлайн-телеметрии](online-telemetry-deployment.md). **Модель доступа — только администратор**
(инструмент пишет файлы на ФС и спавнит процессы, публично его выставлять нельзя).

## Контекст / что уже есть (сверено с кодом)

- **Фронтенд** — `editor.html` + [src/editor/main.ts](../../src/editor/main.ts): чистый
  HTML/TS без зависимостей. Зовёт абсолютные пути `/__editor/*` и открывает игру через
  `window.open('/?game_config=<name>')`.
- **Бэкенд редактора** — это **плагин Vite dev-сервера** `editorDevPlugin()`
  ([src/editor/devPlugin.ts](../../src/editor/devPlugin.ts)) с **`apply: 'serve'`**, то есть он
  существует **только под `vite dev`** и в статический `dist/` **не попадает**. Эндпоинты:
  - `GET  /__editor/game_configs` — список конфигов;
  - `GET  /__editor/game_config/<name>` — файлы конфига;
  - `POST /__editor/save` — **пишет** `src/data/game_configs/<name>/*.json` на диск;
  - `POST /__editor/run-bot` — **спавнит** `sim/bot/run.ts` (через
    `sim/server/node_modules/.bin/tsx`) с `GAME_CONFIG=<name>` и
    `INGEST_URL=process.env.VITE_TELEMETRY_URL`.
- В прод-игру редактор по-прежнему не попадает: статический таргет `editor.html` собирается
  только при `BUILD_EDITOR=1` ([vite.config.ts](../../vite.config.ts)). Для **этого** деплоя
  статика не нужна — мы поднимаем сам `vite dev`, который раздаёт и страницу, и `/__editor/*`.

### Ключевой нюанс: «play this config» требует ребилда игры

[src/data/registry.ts](../../src/data/registry.ts) импортирует все конфиги **статически на
этапе сборки** (eager `import … from './game_configs/<name>/…json'`). Значит:

- **`run bot ▶` работает с живым диском** — `tsx sim/bot/run.ts` читает JSON в рантайме, поэтому
  только что сохранённый конфиг сразу прогоняется и уходит в дашборд.
- **`play this config ▶` для НОВОГО конфига не сработает** на публично собранной игре, пока её
  не пересоберут: бандл знает только те конфиги, что были в `registry.ts` на момент `npm run
  build`. Онлайн-ценность редактора здесь = **правка + прогон ботом + сравнение на дашборде**;
  «поиграть» новым балансом — после ребилда/редеплоя игры (или открыв игру тем же `vite dev`,
  но это уже не публичный билд).

## Подход: приватный `vite dev` за reverse-proxy

Бэкенд редактора неотделим от Vite dev-сервера, поэтому «онлайн-редактор» = **запущенный на
хосте `vite dev`**, закрытый авторизацией и HTTPS на отдельном (непубличном) поддомене/пути.
Переиспользуем `devPlugin` как есть — `save`/`run-bot` работают сразу, доп. кода не нужно.

### 1. Хост с чек-аутом репозитория
Нужен VPS с Node 18+ и **рабочей копией репозитория** (редактор пишет в `src/data/game_configs/`
этого дерева). Поставить зависимости игры (`npm install` в корне) и бэкенд-телеметрии
(`cd sim/server && npm install` — её `tsx` использует `run-bot`).

### 2. Запуск `vite dev` как сервиса
`systemd`-юнит, который держит дев-сервер живым:
```
HOST=127.0.0.1 PORT=5173 VITE_TELEMETRY_URL=https://game.example.com/api npm run dev
```
- `VITE_TELEMETRY_URL` здесь нужен, чтобы **бот-раны из редактора уходили в тот же дашборд**
  (девплагин пробрасывает его как `INGEST_URL`). Это URL уже задеплоенной
  [телеметрии](online-telemetry-deployment.md).
- Слушать локально (`127.0.0.1`), наружу — только через прокси.

### 3. Vite-специфика (обязательно)
- **`server.allowedHosts`** — Vite по умолчанию блокирует запросы с незнакомым `Host`. В
  [vite.config.ts](../../vite.config.ts) (`server`) добавить публичный хост редактора
  (напр. `allowedHosts: ['editor.example.com']`).
- `server.host` уже `true` (слушает на всех интерфейсах) — за прокси можно сузить до
  `127.0.0.1` через env `HOST`.
- `server.watch.ignored` уже включает `**/src/data/game_configs/**` — сохранение из редактора
  не триггерит full-reload (правка сделана отдельно).

### 4. HTTPS + авторизация (модель «только админ»)
- TLS через **Caddy** (авто-Let's Encrypt) или nginx+certbot.
- **Basic-auth** (или токен/OAuth-прокси) на **весь** поддомен/путь редактора — иначе любой
  получит запись файлов и запуск процессов. Опционально сверху IP-allowlist / VPN-only.
- Редактор **не индексируется и не публичен**; держать на отдельном поддомене
  (`editor.example.com`) или закрытом пути.

### 5. Живучесть процесса
`systemd` с `Restart=always` (или контейнер). Падение дев-сервера не должно ронять онлайн-игру
и телеметрию — это отдельный процесс.

## Существующие гард-рейлы в коде (учтены)

- Имя конфига валидируется `NAME_RE = /^[a-z0-9_-]+$/i` в `devPlugin.ts` (нет path-traversal).
- `writeConfig` пишет только whitelist файлов (`GAME_CONFIG_FILES`), не произвольные пути.
- Тем не менее `run-bot` **запускает процесс** → это и есть причина держать всё под админ-авторизацией.

## Персистентность правок

Сохранённые конфиги ложатся в рабочее дерево репо на сервере (`src/data/game_configs/<name>/`).
Чтобы они не потерялись и попали в публичную игру:
- **Закоммитить** изменения из серверного чек-аута в git (ручной шаг или скрипт), затем
  **пересобрать и редеплоить** игру — только тогда `?game_config=<name>` заиграет публично.
- До коммита/ребилда они живут на диске сервера и доступны для `run bot` сразу.

## Минимальный рецепт (Caddy + systemd + basic-auth)

1. VPS, Node 18+, Caddy. Склонировать репо, `npm install` в корне и в `sim/server/`.
2. В `vite.config.ts` добавить `server.allowedHosts: ['editor.example.com']`.
3. `systemd`-юнит:
   ```
   HOST=127.0.0.1 PORT=5173 VITE_TELEMETRY_URL=https://game.example.com/api npm run dev
   Restart=always
   ```
4. `Caddyfile` (basic-auth на весь хост редактора):
   ```
   editor.example.com {
     basic_auth { admin <bcrypt-hash> }
     reverse_proxy 127.0.0.1:5173
   }
   ```
5. Открыть `https://editor.example.com/editor.html`. Править → **save** (пишет на диск
   сервера) → **run bot ▶** (прогон + дашборд телеметрии). Для «поиграть» новым конфигом —
   закоммитить и пересобрать публичную игру.

## Чек-лист «минимум для онлайна»

- [ ] Хост с чек-аутом репо, `npm install` в корне и `sim/server/`.
- [ ] `vite dev` как `systemd`-сервис (`Restart=always`), слушает `127.0.0.1`.
- [ ] `server.allowedHosts` содержит публичный хост редактора.
- [ ] `VITE_TELEMETRY_URL` указывает на задеплоенную телеметрию (для `run-bot`).
- [ ] HTTPS (Caddy/nginx).
- [ ] **Basic-auth/токен** на весь поддомен редактора (+ опц. IP-allowlist/VPN).
- [ ] Понятен флоу персистентности: save → commit → rebuild игры для публичного «play».

## Открытые вопросы

- Где хостить редактор: тот же VPS, что и игра/телеметрия, или отдельный (изоляция дев-сервера)?
- Автокоммит сохранённых конфигов в git (скрипт/кнопка) vs ручной коммит из серверного чек-аута.
- Нужен ли публичный «play this config» вообще, или достаточно «правка + бот + дашборд»
  (тогда ребилд игры не требуется в цикле редактирования).
- Долгосрочно: если приватный `vite dev` в проде нежелателен — портировать `/__editor/*` в
  Fastify `sim/server/` и собирать редактор статикой (`BUILD_EDITOR=1`); более чистый прод,
  но больше работы (см. отвергнутую развилку).
