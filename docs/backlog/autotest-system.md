# Система автотеста и балансировки — спецификация

**Статус:** черновик спецификации (без реализации)
**Версия:** 1.0
**Применимость:** системные игры с уровнями/волнами и внутриигровой экономикой
(Tower Defense, idle, аркада/шмап, рогалик и т.п.)
**Исполнитель правок:** любой кодинг-агент — Claude Code или Cursor (см. §9).

---

## 1. Назначение

Превратить балансировку из ручного «поиграл — кажется, тяжело» в воспроизводимый
конвейер: **прогнать сотни сессий ботом → увидеть по метрикам, где ломается →
сформировать ТЗ на правки → внести правки в код → перепрогнать и проверить, что
починилось.** Система описывает замкнутый агентный цикл и, что важнее, **контракты
между его звеньями** — благодаря им исполнителем последнего шага может быть Claude
Code, Cursor или другой агент без изменения остальной системы.

## 2. Цели и не-цели

**Цели:** детерминированная эмуляция геймплея на реальной логике игры; телеметрия
по сложности, экономике, прогрессии и пейсингу; наглядный дашборд; автоматическое
ТЗ на правки; безопасное замыкание цикла через исполнителя-агента под ревью.

**Не-цели:** замена живого плейтеста (бот меряет играбельность *ботом*, см. §11);
автономные правки без ревью; привязка к конкретному движку или ИИ-инструменту.

## 3. Архитектура цикла

```
        ┌───────────────────────────────────────────────────────────┐
        │                                                           ▼
   ┌──────────┐      телеметрия      ┌────────────┐    aggregate.json   ┌───────────┐
   │   Игра    │ ───────────────────▶ │ Автотест-бот │ ─────────────────▶ │ Дашборд   │
   │ (истина)  │ ◀─── реальная логика │ (headless)  │                    │ (графики) │
   └──────────┘                      └────────────┘                     └───────────┘
        ▲                                  │
        │                                  ▼
        │                            ┌────────────┐   diagnoses
   правки в код/конфиги              │ Анализатор  │ ──────────────┐
        │                            └────────────┘               ▼
   ┌────────────┐  change-request   ┌──────────────────────────────────┐
   │ Исполнитель │ ◀──────────────── │ Генератор ТЗ (change request)     │
   │ (Claude /   │                   └──────────────────────────────────┘
   │  Cursor)    │ ── ревью человеком (HITL) ──▶ применить ──▶ перепрогон
   └────────────┘
```

Все стрелки между звеньями — это **файлы с фиксированными схемами** (§5), а не
вызовы конкретного инструмента. Именно поэтому система переносима.

## 4. Роли и их интерфейсы

| Роль | Ответственность | Вход | Выход |
|---|---|---|---|
| **Игра** | источник истины по механикам | — | импортируемая боевая/экономическая логика |
| **Автотест-бот** | детерминированно эмулирует прохождения | логика игры, сиды, политики | `runs.jsonl` (сырьё на сессию) |
| **Анализатор** | агрегирует и диагностирует | `runs.jsonl` | `aggregate.json` + `diagnoses` |
| **Генератор ТЗ** | оформляет диагнозы как правки | `aggregate.json`, целевые коридоры | `change-request` (§5.3) |
| **Исполнитель** | вносит правки в код в гардрейлах | `change-request` | дифф в коде/конфигах + статус |
| **Дашборд** | визуализирует и сравнивает версии | `aggregate.json` | экран |

Звенья «Игра/Бот/Анализатор/Дашборд» — обычные скрипты и статический HTML; они
инструментально нейтральны по своей природе. Переносимость нужна только для роли
**Исполнитель** — её закрывает контракт `change-request` + гардрейлы (§9–10).

## 5. Контракты данных (ядро переносимости)

Контракты — единственное, на что опираются все звенья и оба исполнителя. Меняешь
схему — обновляешь читателей (дашборд, исполнитель).

### 5.1 Run record (телеметрия на сессию)

Обязательные поля (любая игра):

```jsonc
{
  "seed": 1003,                 // детерминизм: один сид = один забег
  "policy": "greedy",           // политика бота
  "stage": 1,                   // уровень / зона
  "outcome": "victory|defeat",
  "endedAt": { "stage": 1, "wave": 4 }, // где забег завершился
  "durationSec": 132.5,
  "faucets": { "minerals": 740, "enemies": { "drone": 120, "boss": 300 } }, // приток валют по источникам
  "sinks":   { "purchases": [ { "item": "gun_lv3", "atStage": 8 } ] },       // траты/апгрейды и их тайминг
  "entities":{ "bySource": { "drone": { "kills": 210 } } },                  // эффективность сущностей
  "progress":[ { "pct": 0, "metaCurrency": 0 }, { "pct": 100, "metaCurrency": 940 } ] // кривая накопления меты
}
```

Игро-специфичны только *имена* валют/сущностей/апгрейдов; форма — нет.

### 5.2 Aggregate (контракт дашборда)

`aggregate.json` агрегирует `runs.jsonl` по уровням и четырём пластам метрик
(§6): мета (версия баланса, сиды, политики), целевые коридоры, и на каждый
уровень — винрейт, карта «где умирают забеги», метрики волн (бюджет урона,
утечки), потоки валют (приход/трата/итог), эффективность сущностей, тайминг
апгрейдов, кривые меты. Поле `meta.balanceVersion` обязательно — на нём держится
сравнение версий.

### 5.3 Change request (ТЗ на правки) — главный переносимый артефакт

Машиночитаемый и человекочитаемый одновременно. Его одинаково производит
генератор ТЗ и потребляет любой исполнитель:

```jsonc
{
  "id": "cr-2026-06-24-01",
  "basedOn": "sim/out/aggregate.json@<sha>",  // от какого прогона отталкиваемся
  "balanceVersionBefore": "a1b2c3d",
  "summary": "Поздние уровни слишком жёсткие; пушка Lv3 недоступна вовремя",
  "diagnoses": [
    { "metric": "winRate", "where": "stage:6", "observed": 0.22,
      "target": [0.55, 0.70], "severity": "high" },
    { "metric": "upgradeGate", "where": "gun_lv3", "observed": "atStage>=8",
      "target": "atStage~5", "severity": "med" }
  ],
  "changes": [
    { "file": "src/config/waves.ts", "key": "stage6.density",
      "current": 1.0, "proposed": 0.85, "family": "threat",
      "rationale": "снизить плотность волны 6 на 15%",
      "expectedEffect": "winRate stage6 -> в коридор" },
    { "file": "src/config/upgrades.ts", "key": "gun_lv3.cost",
      "current": 1200, "proposed": 950, "family": "economy",
      "rationale": "сделать Lv3 доступным к уровню ~5",
      "expectedEffect": "upgradeGate gun_lv3 -> atStage~5" }
  ],
  "guardrails": { "branch": "balance/cr-2026-06-24-01", "maxDeltaPct": 20,
                  "oneFamilyPerIteration": true, "rerunSeeds": 30,
                  "keepIf": "целевые метрики двигаются к коридору" },
  "status": "proposed|approved|applied|reverted",
  "verification": { "metricBefore": {}, "metricAfter": {}, "decision": "" }
}
```

Каждый пункт `changes` привязан к конкретной метрике из `aggregate.json` — без
этого правка не попадает в ТЗ.

### 5.4 Раскладка в репозитории

```
sim/                      бот, рекордер, агрегатор, политики, дашборд
sim/out/aggregate.json    контракт дашборда (§5.2)
sim/out/runs.jsonl        сырьё (§5.1)
sim/proposals/<id>.md     change request (§5.3), и для людей, и для агента
docs/autotest-system.md   эта спецификация
```

## 6. Таксономия телеметрии (что меряем)

Четыре пласта — ровно то, что на референс-дашборде:

- **Сложность.** Число попыток/прохождений на уровень (где упираются), % побед по
  уровням (кривая сложности), убито врагов.
- **Экономика — краны.** Приток валют по источникам: «из минералов», «из врагов»
  с разбивкой по типам, «всего собрано» (две оси для двух валют).
- **Прогрессия — стоки.** Тепловая карта «после какого уровня куплен апгрейд» по
  компонентам и тирам — когда игрок реально может позволить каждый тир.
- **Пейсинг и мета.** Среднее время сессии по уровням; кривые накопления
  мета-валюты по ходу прохождения (%).

Обязателен минимум на любую игру: исход, точка завершения (уровень/волна), винрейт
и попытки на уровень, длительность сессии, краны по источникам, тайминг стоков,
эффективность сущностей, кривая прогресса. Остальное — игро-специфично.

## 7. Анализ: коридоры и диагнозы

Анализатор сверяет метрики с целевыми коридорами и выдаёт `diagnoses`:

- **Кривая сложности:** где % побед обваливается, а попытки скачут — спайк/стена.
- **Здоровье экономики:** баланс кранов и стоков — голодание или бессмысленное
  накопление валюты.
- **Гейты прогрессии:** апгрейд, который никогда не берут вовремя (дорог/мало
  дохода) или берут мгновенно (дёшев).
- **Пейсинг:** время сессии против целевой полосы.
- **Мёртвый контент:** сущности/апгрейды, почти не влияющие на исход.

Коридоры и линза («впритык при хорошей игре», карта кранов/стоков, балансное
уравнение «урон игрока ↔ EHP») задаются единым источником правил по балансу
(в референс-реализации — скилл `gamejam-spark`), а не на глаз.

## 8. Требования к дашборду

Статический, без серверной части (читает `aggregate.json`; при открытии с диска —
встроенный пример). Панели по четырём пластам: винрейт + тепловая карта сливов;
потоки валют; эффективность сущностей/башен; утечки/убийства и частота
ключевых событий. Поверх — вердикт по целевому коридору, авто-строка рисков и
**сравнение двух версий баланса** (до/после правки) по `meta.balanceVersion`.

## 9. Адаптеры исполнителя (Claude и Cursor)

Исполнитель — это любой агент, который читает `change-request`, вносит правки в
указанные файлы и соблюдает гардрейлы (§10). Ядро системы его не различает.

### 9.1 Под Claude Code

- Сабагент **`balance-sim`** (`.claude/agents/`): прогоняет бота, читает
  `aggregate.json`, пишет `change-request` в `sim/proposals/`. По умолчанию только
  предлагает; авто-тюн (правка `src/config/*`) — по явной команде, в гардрейлах.
- Предзагружен скилл **`gamejam-spark`** (`skills:` в сабагенте) — отсюда коридоры
  и балансная линза.
- Сабагент **`gdd-writer`** синхронизирует ГДД с тем, что наменяли.
- Запуск: делегирование по описанию или явно («Use the balance-sim subagent …»).

### 9.2 Под Cursor

- Правило проекта в `.cursor/rules/balance-sim.mdc` (или `.cursorrules`),
  повторяющее роль и гардрейлы исполнителя и указывающее на контракты:
  «читай `sim/proposals/<id>.md`, вноси только перечисленные `changes`, работай в
  ветке из `guardrails.branch`, не превышай `maxDeltaPct`, меняй одно семейство за
  итерацию, после — запусти `npx tsx sim/runBatch.ts` и сверь метрики».
- Человек открывает нужный `change-request` и просит Cursor его реализовать;
  Cursor читает эту спецификацию для понимания контракта.
- Прогон бота и дашборд — те же скрипты, Cursor их просто запускает.

### 9.3 Любой другой агент

Достаточно, чтобы он умел: прочитать `change-request` (§5.3), отредактировать
указанные файлы, запустить прогон и вернуть `verification`. Никаких других
зависимостей от инструмента нет.

## 10. Гардрейлы и человек в петле

- Правки — всегда на отдельной ветке (`guardrails.branch`), не в рабочем дереве молча.
- Ограниченный шаг: не больше `maxDeltaPct` за итерацию; **одно семейство**
  переменных (угроза ИЛИ доход ИЛИ цены) за проход.
- После правки — перепрогон тем же набором сидов; изменение оставляем, только если
  целевые метрики поехали к коридору, иначе откат.
- Винрейт нельзя «чинить» раздуванием одной сущности — это маскирует дисбаланс.
- ТЗ перед реализацией подтверждает человек (HITL); итоговый мердж в основной
  конфиг — тоже по подтверждению.

## 11. Предусловия и честные границы

- **Детерминизм.** Все источники случайности — через единый seeded RNG; иначе
  версии несравнимы.
- **Замкнутые стоки.** Если траты валют не подключены, панели экономики обманут —
  это чинится первым.
- **Бот ≠ человек.** Система меряет играбельность *ботом по политике*; кривая %
  побед сдвигается, если политика умнеет/глупеет. Держи несколько политик и
  периодически сверяйся с живыми сессиями.
- **Исполнитель под ревью.** Автоправки кода без ветки и подтверждения заставят
  цикл дрейфовать сам от себя.

## 12. Глоссарий

- **Faucet / кран** — источник валюты (килы, минералы, награды).
- **Sink / сток** — трата валюты (установка, апгрейд, реролл).
- **EHP** — эффективное HP врага (HP × брони/резисты/хил).
- **Коридор метрики** — целевой диапазон (например, винрейт 0.55–0.70).
- **Семейство переменных** — группа чисел одного назначения (HP мобов; доход; цены).
- **Change request / ТЗ** — машиночитаемое предложение правок (§5.3).
- **Исполнитель** — кодинг-агент (Claude Code/Cursor/иной), вносящий правки.
